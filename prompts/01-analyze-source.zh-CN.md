2. 找出被选择流量的所有阻塞发送、非阻塞发送、广播、回复、Gossip 和重试路径，以及最窄的共享出站拦截边界和所有更低层旁路；

5. 确定稳定的本地和远端身份，以及它们在接受连接的 Peer、主动拨号的 Peer、断开、移除和重连过程中的行为；

7. 确定现有发送返回值、超时上限、消息大小上限、Peer 生命周期和原生处理器行为对实现施加的约束，以及仓库声明的语言或工具版本、验证配置和原生 CI 或包装命令；
8. 确定每个可独立调度协议角色对应的控制器可见分类，包括共享同一具体原生类型的角色，以及回调解码后能够重新计算各标签的确切原生判别字段；
9. 确定所有可能在普通组装完成后替换或改变选定路由所有者、原型、包装器或编解码器的构造期 Option、自定义 Reactor 机制和注册挂钩，并给出区分无关扩展与不兼容覆盖的证据；
10. 确定规划阶段应检查的最小可能源码范围和聚焦测试。

## 分析纪律

明确区分三类陈述：

- **现有事实（existing fact）**：有源码证据支持的当前文件、符号、调用路径、注册信息或行为；
- **绑定决策（binding decision）**：选定的版本局部边界或映射，并在相关时提供简短理由和可信替代方案；
- **规划约束（planning constraint）**：补丁规划代理必须解决的风险或义务，但不预先规定尚未证明的实现。

不要把计划中的文件、符号、锁、队列或服务表述为现有事实。Prompt 01 可以指出可能的集成范围，但具体补丁结构由 Prompt 02 决定，详细并发加固由 Prompt 04 负责。

追踪足够多的调用者和被调用者，以证明覆盖完整性。尤其不能在只找到一种发送模式、一个连接方向或最终原生处理器后就停止。同时，一旦共享边界已经得到证明，就应避免继续进行穷举式叙述。

对于回调再注入，应确定：接受前必须验证什么、哪些稳定数据可以跨越异步交接、哪些与连接相关的状态必须在分发时重新解析。记录相关风险，例如原生处理器同步停止其来源 Peer、关闭过程与已接受工作发生竞争，或者注册表发生变化。除非源码证明不存在合理的合规实现，否则这些都属于规划约束。

回调返回 `2xx` 只表示验证以及规范定义的原生交接或有界队列交接成功。它不得等待或宣称协议处理器完成、应用工作完成、数据持久化或协议状态发生变化。接受后的任何可能丢失都必须作为“接受后完整性失败”暴露给后续规划，而不能静默丢弃。

对于出站调用，保留阻塞和非阻塞 API 之间的区别。确定每条路径上已有的工作和边界，并标记任何可能改变返回行为的拟议状态查找、计数器、锁、序列化或队列操作。具体同步算法属于后续阶段。

不要假设一个具体原生类型只对应一个控制器可见 `Type`。当共享同一原生类型的多个协议角色可以被独立调度时，必须绑定其原生判别字段并记录彼此不同的必需标签。对于 CometBFT 投票，应从源码确定如何在不要求控制器解释不透明适配器载荷的前提下区分 Prevote、Precommit 及其 `nil` 形式。

应在所有普通构造 Option 已应用之后分析注册关系。如果自定义 Reactor 或扩展挂钩能够替换选定路由的所有者，应记录原始所有者、替换机制、启动时可用的兼容性证据，以及最安全的合规策略是验证还是显式拒绝。非空路由注册本身不构成兼容性证据。

只检查足以确定仓库声明信息的构建、CI、pre-commit 和验证配置，包括语言版本、非标准工具版本、配置文件及原生命令入口。将缺少版本固定也作为事实记录。不要在此阶段安装工具、选择主机特有路径或测试本地工具可用性；环境兼容性和可执行规划属于 Prompt 02。

不要实现或设计生产控制器。后续验证阶段可以添加仅用于测试的假控制器。

## 阶段边界与状态

当所有强制绑定都有源码证据支持，并且至少存在一条合理的实现路径时，返回 `PASS`。将正常的实现风险记录给规划、生成、加固和验证阶段；不要把每个尚未解决的编码细节都变成阻塞项。

只有在以下情况下返回 `BLOCKED`：

- 上游阶段门失败；
- 规范互相矛盾，或者过于含糊而无法完成绑定；
- 无法在不猜测的情况下确定消息清单、出站覆盖、稳定身份、原生再注入入口、禁用模式保持、生命周期所有者或致命错误路径；
- 现有源码证据排除了合规实现；
- 需要额外的范围外源码。

因范围受阻时，返回 `AGENTS.md` 要求的最小 `SCOPE_EXPANSION_REQUEST`。

## 输出

编写紧凑的 `01-binding.json`，包含：

- artifact 元数据、规范指纹、源码 revision、初始工作区状态、`status` 和 `downstream_allowed`；
- 分组后的 `selected_messages` 和 `excluded_messages`，包括协议角色、原生类型、控制器可见标签及原生判别字段、route、所有者、wrapper 和简明证据引用；
- `outbound_binding`，包括已覆盖的发送模式、选定边界和旁路风险；
- `inbound_binding`，包括普通接收链和候选再注入入口；
- `callback_contract`，包括验证/接受边界、入队的稳定数据、分发时查找和接受后责任；
- `identity_and_lifecycle`，包括身份、Peer 替换、配置、就绪、关闭和致命错误传播绑定；
- `extension_compatibility`，包括构造期替换挂钩、预期的选定路由所有者与注册、无关扩展行为、启动检测证据和仍未解决的兼容性限制；
- `repository_toolchain`，包括仓库声明的语言和验证工具版本、版本或配置证据、原生命令入口，以及明确列出的未固定版本工具；
- `implementation_constraints`，只包含从源码推导、供后续阶段处理的义务和风险；
- `candidate_surface`，包括可能涉及的现有文件和符号，但不虚构详细新增项；
- `test_anchors` 和少量可执行的仓库原生验证命令；
- 去重后的 `evidence`、`files_read`、重要 `commands`、`assumptions`、`unresolved_risks` 和 `scope_expansion_requests`。

共享同一 route、所有者、wrapper 和证据的消息应放在一组中，不要为每个协议角色重复一个大型对象。使用简短证据标识符，并只在一处将其映射到 `path:line` 位置。只记录对建立绑定或验证 artifact 有实质作用的命令；每条命令都必须是可直接执行的字面命令，不能是自然语言伪命令。

`01-analysis-notes.md` 只用于记录简短理由、被否决的边界替代方案，以及不便放入 JSON 的风险。不要重复 JSON 内容，也不要复述规范。应优先生成一份可供下游代理使用的小型源码映射，而不是详尽的研究日志。
