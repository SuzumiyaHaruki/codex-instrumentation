## 规划任务

生成一份最小且有序的计划，使其能够在保留固定绑定的同时实现所有适用的规范要求。

每个实施步骤都必须记录：

- 目的和所属阶段（`functional` 或 `refinement`）；
- 绑定引用、规范条款和现有源码证据；
- 需要读取的确切文件，以及需要新增或修改的确切文件；
- 涉及的现有符号，以及清楚标注为新增或修改的计划符号；
- 设计决策、重要备选方案，以及所选方案为何最小；
- 后续步骤必须保持的依赖关系和不变量；
- 聚焦测试和可执行的验证命令；
- 所控制的风险，以及回滚行为或失败行为。

这些有序步骤必须构成有效的依赖图。一个步骤只能引用现有符号，或由其声明的前置步骤创建的计划符号。每个功能阶段检查点都必须能够独立编译。如果两项修改相互依赖，无法形成有效的中间构建，则应将它们放入同一个明确命名的 `atomic_group`，并且只在该原子组全部完成后运行检查；不要把明知无法编译的中间步骤描述为可以独立验证。

处理 Prompt 01 中的每一项 `implementation_constraint` 和 `unresolved_risk`。每一项都必须映射到以下类别之一：

- 一个功能实现步骤；
- 一个并发或生命周期完善步骤；
- 一个聚焦验证步骤；或
- 一个阻塞原因。

不要只是将未解决风险复制到新产物中。必须说明后续工作需要实现什么或证明什么。

计划必须明确建立以下内容：

- 从原生包装或序列化到提交中央控制器的精确转换位置，并且不得回退到原始发送路径；
- 可重新计算的控制器可见分类，并保留目标剖面要求的每个可调度协议角色；对于 CometBFT，Prevote、Precommit 及其 `nil` 投票不得折叠为一个通用标签；
- 在所有 Option 应用后、就绪前执行兼容性门禁，验证选定路由的所有者、原型、包装器和编解码器，拒绝未经证明兼容的选定路由替换，并允许无关的自定义 Reactor；
- 保留完整的常规接收边界，或以证据证明行为等价的边界替代它，而不是直接调用最终处理器；
- 回调移交数据必须不可变，并在分发时重新解析与连接状态有关的信息；
- 非阻塞发送保持非阻塞；阻塞发送、队列、HTTP、重试和回调请求体均具有边界；
- 每个有向节点对拥有唯一 ID，回调保持顺序和多重投递语义，且本地不进行去重；
- 在目标消息发送前达到就绪状态，安全回滚部分启动过程，实现幂等关闭，并对已接收工作进行核算；
- 按规范条款准确处理消息拒绝、同步拒绝、不变量失败、控制器故障，以及有序且非零退出的致命错误传播；
- 关闭插桩模式时的行为、非目标流量、协议规则、消息模式、持久化格式、路由和超时语义保持不变。

对于每一个配置或硬编码的资源限制——包括队列、保留字节数、请求体、重试预算、工作线程以及身份或计数器注册表——都必须说明达到上限时会发生什么，并将该结果映射到规范定义的拒绝类别或致命失败类别。不要引入一种永久容量限制，使原本有效的实验流量可能被拒绝，却没有定义该限制的生命周期、恢复方式和诊断行为。

测试必须先存在，步骤才能声称运行它们。只有当相应测试文件和符号已由同一步骤、同一原子组或某个已声明的前置步骤创建后，使用 `go test -run` 指定新计划测试的命令才有效。在依赖过滤测试命令之前，应规划类似 `go test <pkg> -list '<pattern>'` 的测试发现检查，并断言它至少报告一个匹配的测试符号（普通包状态输出不计）；退出码为零但没有匹配测试的执行结果不能作为证据。

将语言或构建工具链以及每个非标准验证可执行程序视为实验的版本化输入。优先采用当前检出仓库在模块、CI、pre-commit、构建或工具配置中固定的精确版本；不要假设 `PATH` 中最先出现的多个可执行程序彼此兼容。记录版本证据、可用性探针、预期版本输出，以及运行命令前所需的环境或仓库原生引导方式。包含会影响分析的兼容关系，例如 linter 读取编译器 export data。在本阶段探测计划采用的实际执行形式。某个二进制即使存在，只要它拒绝仓库配置或无法读取所选语言工具链产生的产物，在规划意义上仍属于不可用。不要仅为适配更新的本地工具而修改仓库配置。

将必需的环境准备和版本或兼容性探针放入 `verification_commands`，并排在首个依赖该工具的命令之前；也可以将环境作为该命令的字面前缀。仅在 `verification_toolchain` 中用文字说明不能算作可执行的环境准备。后续 Agent 必须能够在不继承本阶段交互式 shell 状态的情况下复现选定的可执行程序。

如果仓库没有固定版本，应优先使用其原生包装命令（如存在）。否则，将实验运行环境提供的版本记录为明确的可复现性假设，并加入兼容性探针。如果某项必需外部工具没有经过验证的兼容执行路径，应在规划阶段阻塞，而不是等到 Prompt 03 或 Prompt 04 才首次发现。工具准备必须位于生产代码之外，也不得为目标系统新增依赖。

仅在测试需要时，才可在新增测试文件中包含轻量级假控制器。绝不规划生产级控制器、调度器、策略存储、模型映射器、状态观测器或模糊测试引擎。

聚焦测试计划必须分别覆盖协议角色标签区分和回调标签重新计算，而不能只测试原生类型往返。它还必须覆盖每个选定原生 Reactor 所有者的替换以及至少一个无关自定义 Reactor，证明前者启动失败而后者保持兼容。

功能阶段不必证明所有罕见交错执行或性能极限，因为 Prompt 04 会完善并发和生命周期行为。但是，功能阶段不得明知故犯地引入不安全的所有权模型、无界资源、已接收工作静默丢失、直接发送回退或协议语义变化。应明确分派可在后续完善的风险，而不是假装这些风险已经解决。

## 状态判定

当绑定仍然有效、所有固定结果都得到保留、每一项强制要求和已记录风险都在计划中有明确负责人，并且功能阶段具备连贯可行的实施路径时，返回 `PASS`。

出现以下任一情况时，返回 `BLOCKED`：

- 上游门禁、哈希、源码版本或关键证据检查失败；
- 必须修正绑定；
- 某项强制风险不存在合理且合规的设计；
- 完成规划必须重新开展广泛源码探索或读取范围外文件；
- 唯一可见的实现方式会修改受保护的协议行为或其他禁止修改的部分；
- 步骤依赖图存在未解决的符号依赖、明知无法编译的检查点，或尚不能发现其指定测试的测试命令；或
- 某项必需语言或验证工具没有与仓库兼容的版本组合、可用性探针或可复现执行路径；或
- 计划试图推迟已知的根本安全性或正确性故障，而不是可在后续完善的实现风险。
