# 中央控制器介导的消息投递：核心规范

## 1. 目的

本规范为分布式系统定义一种可复用的插桩模式。在该模式下，选定的节点间消息不再通过原有的直接网络路径投递。发送节点将消息提交给外部中央控制器；只有当且仅当控制器将消息发送到目标节点的回调端点时，该消息才可以进入目标节点的原生接收路径。

控制器可以投递、延迟、重排、复制或永久丢弃已提交的消息。控制器本身及任何控制器侧策略的实现均不属于节点改动范围。

本规范约束的主要成果是由基于 LLM 或等价的自动实现智能体依据要求和源码分析生成的节点侧插桩。实现范围包括源码发现、补丁生成、控制器提交、回调接收、原生重新注入和验证；不包括控制器策略、模型引导调度、协议状态观测、形式化模型映射或覆盖率引导。

本文件与具体系统无关。它定义所有受支持系统共同遵守的稳定语义契约、线协议契约、故障模型、实现证据和一致性验证义务。`target-*.md` 目标规范负责将该契约绑定到一个目标系统。本文件与适用的目标规范均为规范性文件。生成的插桩 **SHOULD** 能够供未来的外部调度器或模型引导测试系统使用，但插桩任务本身不需要实现这些系统。

## 2. 规范性用语

术语 **MUST**、**MUST NOT**、**REQUIRED**、**SHALL**、**SHALL NOT**、**SHOULD**、**SHOULD NOT** 和 **MAY** 应按照 RFC 2119 与 RFC 8174 的定义解释。

本文件中的要求使用 `CORE-*` 形式的稳定标识符。目标规范可以强化核心要求，但 **MUST NOT** 弱化或违背核心要求。

## 3. 规范套件结构

规范套件包含两个规范性层次：

| 层次 | 职责 | 稳定性 |
| --- | --- | --- |
| **核心规范** | 定义与具体系统无关的介导语义、控制器线协议、故障分类、安全约束和公共一致性属性。 | 在不同目标系统间原样复用。 |
| **目标规范** | 定义目标系统术语、选定消息域、代码库分析义务、原生绑定契约和目标专属一致性证据。 | 面向一个系统族编写，并依据当前检出的版本完成解析。 |

目标规范是一份实现剖面（realization profile），而不是特定版本的补丁说明。它规定实现者 **MUST** 发现、保持、绑定和证明的事项。除非某些文件、符号或已注册消息类型属于目标系统稳定的公共架构，否则其具体取值应从当前源码版本中推导。

每一份 `target-*.md` 规范 **MUST** 使用以下顶层剖面。允许增加子节，但各节职责 **MUST** 保持分离，并且 **MUST** 按以下顺序出现。

| 章节 | 必需内容 |
| --- | --- |
| **1. 规范剖面** | 目标身份、核心规范依赖、要求命名空间、版本策略和规范性状态。 |
| **2. 语义范围** | 目标专属目标、协议或子系统术语、受保护语义和明确的非目标。 |
| **3. 代码库绑定分析** | 发现发送、接收、身份、路由、编解码、生命周期、绕行路径以及生成机器可读插桩清单时需要提供证据的分析义务。 |
| **4. 受控消息域** | 被选择或排除的语义消息族，以及推导具体消息清单的权威方法。 |
| **5. 目标绑定契约** | 将核心身份、路由、载荷、类型标签和原生投递映射到目标抽象。 |
| **6. 集成约束** | 插入位置、局部性、兼容性、并发和依赖约束。 |
| **7. 配置与生命周期** | 启用、就绪、注册、启动、关闭和资源所有权。 |
| **8. 故障映射** | 将目标专属故障归类到核心错误分类。 |
| **9. 一致性验证** | 与条款关联的静态、单元、集成、竞态和回归证据。 |
| **10. 验收轨迹** | 覆盖投递、省略、重排、复制、格式错误输入、基础设施故障和普通流量的端到端可观察场景。 |

- **CORE-PROFILE-001:** 目标剖面在提出实现要求前，**MUST** 标识目标系统或系统族、适用核心规范、要求命名空间和源码版本策略。
- **CORE-PROFILE-002:** 目标专属规范性要求 **MUST** 使用稳定标识符。解释性的协议背景 **MAY** 不编号，但 **MUST NOT** 隐式引入实现义务。
- **CORE-PROFILE-003:** 语义范围 **MUST** 使用目标系统的原生术语，同时区分协议层概念与特定版本的实现名称。
- **CORE-PROFILE-004:** 代码库绑定分析 **MUST** 要求源码级证据，并且在生产代码修改完成前产出集成映射、具体选定消息清单和绕行风险清单。
- **CORE-PROFILE-005:** 受控消息域 **MUST** 主要依据语义归属或协议角色定义选择范围。历史数值标识符或类型列表 **MAY** 用于说明，但 **MUST NOT** 取代从当前版本进行的权威发现。
- **CORE-PROFILE-006:** 目标绑定契约 **MUST** 说明每个核心身份、路由、序列化、回调和原生投递概念如何映射到目标系统，同时不得重新定义原生消息语义。
- **CORE-PROFILE-007:** 故障映射 **MUST** 按第 5.4 节对目标专属故障进行归类。它 **MUST NOT** 发明会弱化核心禁止回退规则或致命完整性规则的恢复行为。
- **CORE-PROFILE-008:** 一致性验证 **MUST** 将目标要求关联到可观察证据。没有对应条款和验证义务的代码改动是不完整的。
- **CORE-PROFILE-009:** 代码库绑定分析 **MUST** 生成一份机器可读的插桩清单，记录 `CORE-PATCH-006` 至 `CORE-PATCH-009` 定义的已发现绑定关系和生成改动。该清单用于支持自动化和下游适配器；它 **MUST NOT** 取代源码检查或一致性测试。

## 4. 术语

| 术语                                          | 含义                                                   |
| --------------------------------------------- | ------------------------------------------------------ |
| **原始传输（original transport）**      | 系统现有的节点间直接消息路径。                         |
| **控制器（controller）**                | 接收选定消息并决定是否以及何时转发这些消息的外部服务。 |
| **插桩节点（instrumented node）**       | 启用了控制器介导投递的节点。                           |
| **选定消息（selected message）**        | 由目标规范归类为需经控制器介导的消息。                 |
| **未选定消息（unselected message）**    | 未被归类为选定消息的任何消息。                         |
| **源身份（source identity）**           | 逻辑发送方的稳定网络身份。                             |
| **目标身份（destination identity）**    | 预期接收方的稳定网络身份。                             |
| **回调端点（callback endpoint）**       | 插桩节点上用于接收控制器转发消息的 HTTP 端点。         |
| **原生接收路径（native receive path）** | 普通网络解码完成后调用的现有内部处理器。               |
| **重新注入接受（reinjection acceptance）** | 控制器转发消息成功完成验证，并被交给原生接收路径或其有界输入队列；它不表示协议已经执行或状态已经改变。 |
| **插桩清单（instrumentation manifest）** | 针对一次插桩结果，以机器可读形式记录源码绑定、选定消息清单、生成钩子、配置和验证证据。 |

## 5. 必需架构

对于每一条选定消息，可观察到的路径 **SHALL** 为：

```text
source node
    -> controller submission endpoint
    -> controller policy decision
    -> destination node callback endpoint (only if delivered)
    -> destination native receive path
```

禁止使用的路径为：

```text
source node -> destination node through the original transport
```

未选定消息继续使用原始传输。

### 5.1 运行模式

- **CORE-MODE-001:** 插桩 **MUST** 由用户显式选择启用，并且默认禁用。
- **CORE-MODE-002:** 插桩禁用时，消息路由、启动、关闭、配置默认值、返回值和错误行为 **MUST** 保持不变。
- **CORE-MODE-003:** 启用插桩 **MUST NOT** 改变分布式协议的有效性规则、法定人数规则、状态转换、消息模式、持久化格式或超时语义。
- **CORE-MODE-004:** 实现中 **MUST NOT** 包含控制器策略。节点只负责提交选定消息和接收控制器转发的消息。

### 5.2 完整拦截

- **CORE-INT-001:** 目标规范 **MUST** 无歧义地定义选定消息集合。
- **CORE-INT-002:** 所有能够发送选定消息的出站 API，包括阻塞、非阻塞、广播、重试和回复路径，均 **MUST** 被覆盖。如果另一种发送方法可以绕过拦截，则仅包装一种发送方法并不充分。
- **CORE-INT-003:** 一旦消息被归类为选定消息，节点在任何情况下都 **MUST NOT** 通过原始传输发送该消息。
- **CORE-INT-004:** 编码失败、插桩队列已满、控制器不可用、超时、非成功 HTTP 响应或关闭过程，均 **MUST** 按照第 5.4 节处理，并且 **MUST NOT** 导致回退到原始传输。
- **CORE-INT-005:** 在目标节点的回调端点接受控制器转发的该消息实例之前，选定消息 **MUST NOT** 进入目标节点的原生接收路径。
- **CORE-INT-006:** 启用插桩时，通过原始网络接收路径到达的选定消息 **MUST NOT** 被投递到原生接收路径。实现 **MUST** 拒绝或忽略该消息，并提供诊断信息。
- **CORE-INT-007:** 未选定消息 **MUST** 继续通过原始传输发送，并且 **MUST NOT** 被提交给控制器。

### 5.3 发送结果语义

- **CORE-SEND-001:** 选定消息的阻塞发送 **MAY** 等待，但等待时间只能位于系统现有发送时限或更严格的已配置时限内。
- **CORE-SEND-002:** 选定消息的非阻塞发送 **MUST** 保持非阻塞。它 **SHOULD** 将消息放入有界的控制器出站队列；当队列无法接收消息时，应立即返回失败。
- **CORE-SEND-003:** 成功的发送结果表示插桩层已接受提交责任；它并不表示控制器将会投递该消息。
- **CORE-SEND-004:** 如果在接受消息之前已经得知选定消息发送失败，该失败 **MUST** 通过现有返回值或错误机制暴露并记录到诊断信息中。失败 **MUST NOT** 被转换为直接发送。
- **CORE-SEND-005:** 如果提交在原始调用返回成功后失败，则插桩实验结果已不再可信，节点 **MUST** 进入 `CORE-ERR-008` 定义的致命故障处理流程。

### 5.4 错误分类与必需处理方式

实现 **MUST** 区分单条无效消息与已经损坏的插桩路径。在基础设施或不变量故障后静默继续会使实验结果不可靠；而因每一条格式错误的输入都终止进程，则会使回调端点不必要地脆弱。

- **CORE-ERR-001:** 每个插桩错误 **MUST** 被且仅被归类为以下一种：消息级拒绝、同步发送拒绝、本地不变量故障或控制器/控制通道故障。
- **CORE-ERR-002:** 格式错误、过大、目标错误、来源未知、路由未知或其他无效的控制器转发消息属于消息级拒绝。节点 **MUST** 只拒绝或丢弃该消息实例，返回合适的非 `2xx` 响应，生成结构化错误和指标，并继续运行。
- **CORE-ERR-003:** 有界本地背压，或在有序关闭已经开始后尝试发送，只要在消息被接受前向调用者报告，均属于同步发送拒绝。发送操作 **MUST** 返回失败并记录该事件，节点保持运行或继续其现有关闭流程，同时 **MUST NOT** 使用原始传输。
- **CORE-ERR-004:** 无法对合法的选定出站消息进行分类或编码、合法选定消息缺少必需的本地解码器或原生路由、内部队列损坏，或违反拦截不变量，均属于本地不变量故障。节点 **MUST** 将其视为致命故障，而不能静默丢弃消息。
- **CORE-ERR-005:** 启动就绪后必需控制器路径发生故障，包括连接被拒绝、请求超时、HTTP 行为异常、提交收到非 `2xx` 响应或回调监听器意外终止，均属于控制器/控制通道故障，并且 **MUST** 被视为致命故障。
- **CORE-ERR-006:** 节点声明就绪前，注册 **MAY** 使用有界重试。如果重试预算耗尽，启动 **MUST** 失败。节点 **MUST NOT** 以未注册或部分就绪状态加入实验。
- **CORE-ERR-007:** 选定消息已被报告为发送成功后发现的任何异步丢失都 **MUST** 被视为致命故障，因为调用者已经无法观察或补偿该丢失。
- **CORE-ERR-008:** 致命故障处理 **MUST** 以原子方式将插桩标记为不健康，停止接受新的选定消息发送，记录确切原因，启动节点的有序关闭，并使所属进程返回非零失败状态。库代码 **SHOULD** 返回或传播带类型的致命错误；在可以有序关闭时，**MUST NOT** 使用不可控 panic 或突然退出进程。
- **CORE-ERR-009:** 重复发生的消息级拒绝 **MAY** 触发由操作者配置的实验中止阈值，但单条无效转发消息 **MUST NOT** 终止节点，除非处理该消息时暴露了本地不变量故障。

## 6. 控制器 HTTP 契约

控制器地址以及回调监听/通告地址 **MUST** 为配置值。除非目标规范另有明确规定，地址采用 `host:port` 格式，并且节点在隔离的测试网络中使用明文 HTTP。

本节中的所有请求 **MUST** 使用 `Content-Type: application/json`。

### 6.1 节点注册

- **CORE-REG-001:** 插桩节点在向控制器注册之前 **MUST** 已暴露其回调端点。
- **CORE-REG-002:** 节点在能够提交第一条选定消息之前 **MUST** 完成注册。
- **CORE-REG-003:** 注册使用 `POST http://<controller>/replica`，请求体为以下 JSON 对象：

```json
{
  "id": 1,
  "alias": "stable-source-identity",
  "addr": "node-callback-host:port"
}
```

- **CORE-REG-004:** `id` 是实验分配的整数节点编号；`alias` 是消息路由使用的稳定网络身份；`addr` 是控制器可访问的地址。
- **CORE-REG-005:** 只有当控制器路由确实需要额外的非秘密身份信息时，目标适配器才 **MAY** 添加 `metadata` 对象。目标规范或实现报告 **MUST** 说明其字段。私钥和签名秘密 **MUST NOT** 被注册。
- **CORE-REG-006:** 只有收到 `2xx` 响应后注册才算成功。响应体不具有语义意义。
- **CORE-REG-007:** 节点只 **MAY** 在就绪前使用有界且感知关闭状态的退避策略重试失败的注册。重试预算耗尽时，插桩启动 **MUST** 按照 `CORE-ERR-006` 失败。
- **CORE-REG-008:** 对于不经过可选同步阶段的节点，注册也 **MUST** 正常工作。就绪状态 **MUST** 与回调和发送路径的实际就绪状态绑定，而不能绑定到某个偶然发生的协议转换。

### 6.2 已提交和已转发消息

两个方向使用相同的外层 JSON 对象：

```json
{
  "From": "source-identity",
  "To": "destination-identity",
  "Data": "base64-encoded-target-payload",
  "Type": "target-defined-message-label",
  "ID": "source-identity_destination-identity_0"
}
```

- **CORE-WIRE-001:** 源节点使用 `POST http://<controller>/message` 提交该对象。
- **CORE-WIRE-002:** 目标节点使用 `POST http://<node-callback>/message` 接收控制器转发的对象。
- **CORE-WIRE-003:** JSON 成员名称及大小写 **MUST** 严格为 `From`、`To`、`Data`、`Type` 和 `ID`。
- **CORE-WIRE-004:** `From` 和 `To` **MUST** 包含稳定网络身份，而不是套接字地址或临时连接索引。
- **CORE-WIRE-005:** `Data` 是使用标准 JSON base64 编码表示的不透明字节序列。其解码后内容由目标规范定义。控制器无需解码或修改它。
- **CORE-WIRE-006:** `Type` 是目标规范定义的确定性、人类可读分类器。它是控制器元数据，并且 **MUST NOT** 替代 `Data` 内目标系统的原生路由元数据。
- **CORE-WIRE-007:** `ID` **MUST** 由源节点按照 `<From>_<To>_<counter>` 格式生成。计数器从 `0` 开始，对该有向源/目标节点对中的每条已提交消息递增一，并且在并发发送时保持安全。
- **CORE-WIRE-008:** 源节点 **MUST** 在提交前分配 `ID`。目标节点 **MUST** 保留该 ID 用于诊断，但 **MUST NOT** 在本地按 ID 去重；控制器有意进行的消息复制必须保持可观察。
- **CORE-WIRE-009:** 只有收到 `2xx` 响应后，提交才算被接受；提交失败按照 `CORE-ERR-005` 处理。只有在完整外层对象和目标载荷均已验证，并安全交给原生接收路径或其有界本地重新注入队列后，回调才返回 `2xx`。
- **CORE-WIRE-010:** 格式错误的请求 **MUST** 返回 `400`。格式正确但对当前节点或当前连接状态无效的请求 **MUST** 返回合适的非 `2xx` 响应、**MUST NOT** 被投递，并按照 `CORE-ERR-002` 作为消息级拒绝处理。
- **CORE-WIRE-011:** `ID` 标识已提交的消息实例，并且从源节点提交到控制器每一次回调都 **MUST** 原样保留。控制器有意复制消息时，重复回调保留相同 `ID`；节点 **MUST** 将每次回调视为独立的重新注入尝试，并且 **MUST NOT** 通过修改 `ID` 推导单独的投递标识。
- **CORE-WIRE-012:** 回调 `2xx` 响应和重新注入接受只表示插桩验证和原生交接成功。它们 **MUST NOT** 被报告或解释为协议处理器已经完成、在语义上接受消息、改变协议状态、持久化数据或执行形式化模型动作的证据。

## 7. 目标侧投递

- **CORE-RECV-001:** 回调处理器 **MUST** 验证 `To` 等于本地稳定身份。
- **CORE-RECV-002:** 回调处理器在将消息入队之前 **MUST** 验证 `From`、`Type` 以及所有由目标规范定义的路由字段和载荷字段。
- **CORE-RECV-003:** 实现 **MUST** 将 `From` 解析为原始接收路径本应提供的同一个逻辑 Peer 对象或源上下文。来自未知或已断开连接源的消息 **MUST NOT** 被投递。
- **CORE-RECV-004:** 目标载荷 **MUST** 使用系统现有的编解码器以及包装/解包装规则解码。插桩层 **MUST NOT** 定义另一套具有消息语义的格式。
- **CORE-RECV-005:** 成功解码后，节点 **MUST** 使用原始路由元数据、逻辑来源和已解码消息调用现有原生接收路径。
- **CORE-RECV-006:** 控制器转发的投递 **MUST** 经过与正常接收的等价消息相同的语义验证和 Peer 错误处理行为。格式错误的控制器输入 **MUST** 只拒绝该消息，并且 **MUST NOT** 导致 panic 或停止进程，除非它暴露了本地不变量故障。
- **CORE-RECV-007:** 消息 **MUST** 按回调接受顺序投递。节点 **MUST NOT** 恢复源发送顺序、去重或添加独立延迟。这些决策属于控制器。
- **CORE-RECV-008:** 队列访问、身份查找、计数器和生命周期状态 **MUST** 保持并发安全。队列 **MUST** 有界，或提供等效的背压机制。
- **CORE-RECV-009:** 插桩职责在调用原生接收路径，或将消息安全放入该路径的队列后结束。补丁 **MUST NOT** 修改协议处理器来制造执行确认、推断状态转换，或仅为未来测试集成而暴露协议状态修改。
- **CORE-RECV-010:** 结构化诊断信息 **MUST** 使相同的 `ID`、`From`、`To` 和 `Type` 能够在提交、回调拒绝、重新注入接受和重新注入失败之间建立关联。诊断名称 **MUST** 区分重新注入接受与协议执行，并且 **MUST NOT** 声称插桩层无法观测的效果。

## 8. 生命周期与故障处理

- **CORE-LIFE-001:** 启动顺序 **MUST** 为：验证配置、绑定回调监听器、启动投递处理、向控制器注册，随后才允许提交选定消息。
- **CORE-LIFE-002:** 绑定失败、控制器配置无效或注册重试耗尽 **MUST** 明确导致插桩启动失败。就绪后必需的回调或控制器路径发生故障 **MUST** 触发有序的致命关闭。
- **CORE-LIFE-003:** 关闭过程 **MUST** 停止新的提交、停止接受回调请求、取消重试和工作任务，并关闭原始传输。
- **CORE-LIFE-004:** 关闭操作 **MUST** 具有幂等性；即使启动只完成了一部分，也 **MUST NOT** panic。
- **CORE-LIFE-005:** 任何工作任务都不得忙等。投递 **SHOULD** 由事件驱动；如果无法避免轮询，则 **MUST** 使用合理且感知关闭状态的间隔。
- **CORE-LIFE-006:** HTTP 客户端 **MUST** 设置有限的拨号、响应头和总请求超时。响应体 **MUST** 被关闭。
- **CORE-LIFE-007:** 重新连接 **MUST** 更新源身份映射，断开连接 **MUST** 删除过期映射，使延迟消息无法归因于已经失效的连接。

## 9. 安全边界

- **CORE-SEC-001:** 本模式用于受控实验。除非目标规范增加身份认证和传输安全，否则文档中 **MUST** 明确说明本模式在不可信网络中不安全。
- **CORE-SEC-002:** 回调监听器 **SHOULD** 绑定到受限网络接口，并且网络访问 **SHOULD** 仅限控制器。
- **CORE-SEC-003:** 注册和消息提交 **MUST NOT** 暴露私钥、签名秘密、认证令牌或无关的节点状态。
- **CORE-SEC-004:** 日志 **MUST NOT** 输出秘密或完整的敏感载荷。
- **CORE-SEC-005:** 请求体大小和解码后载荷大小 **MUST** 受目标系统现有最大消息约束限制。

## 10. 补丁纪律

- **CORE-PATCH-001:** 改动 **MUST** 保持最小化，并局限于传输、节点装配、配置和聚焦测试附近。
- **CORE-PATCH-002:** 现有消息类型、路由标识符、编解码器、协议处理器和持久化数据格式 **MUST** 被复用，而不能替换。
- **CORE-PATCH-003:** 普通节点实现 **SHOULD** 被共享。当传输工厂、选项或小型条件分支能够提供插桩模式时，禁止大段复制节点构造逻辑。
- **CORE-PATCH-004:** 当语言运行时或现有依赖已经提供 HTTP 或 JSON 能力时，不得仅为实现这些部分而新增生产依赖。
- **CORE-PATCH-005:** 为适配插桩，现有测试 **MUST NOT** 被删除、跳过、弱化或重写。
- **CORE-PATCH-006:** 实现流程 **MUST** 生成一份机器可读的插桩清单。当前阶段提示确定其可写位置；如果未授权清单写入路径，则 **MUST** 在最终报告中包含等价 JSON 对象，而不是写入范围之外的位置。
- **CORE-PATCH-007:** 清单 **MUST** 记录目标与源码版本、选定和明确排除的消息族、已发现的出站钩子与绕行风险、回调与原生重新注入绑定、身份/路由/编解码/配置/生命周期绑定、已修改文件和符号及其规范条款、验证命令与结果、假设以及未解决缺口。无法确定的项目 **MUST** 标记为未解决，而不能猜测。
- **CORE-PATCH-008:** 清单 **MUST NOT** 包含秘密材料，**MUST** 根据生成补丁和源码证据进行核对，并且 **MUST NOT** 包含或要求控制器策略、协议状态观测、形式化模型动作、模型状态或模糊测试策略。

清单至少具有以下 JSON 结构：

```json
{
  "schema_version": 1,
  "target": {
    "name": "target-system",
    "revision": "source-revision"
  },
  "selected_messages": [],
  "excluded_messages": [],
  "bindings": {
    "outbound_hooks": [],
    "bypass_risks": [],
    "callback": {},
    "native_reinjection": {},
    "identity": {},
    "routing": {},
    "codec": {},
    "configuration": {},
    "lifecycle": {}
  },
  "changes": [],
  "verification": [],
  "assumptions": [],
  "unresolved_gaps": []
}
```

- **CORE-PATCH-009:** 清单生成方 **MAY** 添加目标专属成员，但 **MUST NOT** 重命名、删除或改变最小成员的含义。每个绑定和改动条目 **MUST** 包含源码证据或明确的未解决标记，并且每个值都 **MUST** 是合法 JSON 数据。

## 11. 必需验证

实现 **MUST** 添加聚焦测试，证明以下全部事项：

1. 插桩默认关闭，并且普通路径保持不变。
2. 每一条选定消息的阻塞、非阻塞、广播、回复和重试发送路径都会到达控制器，而不会到达直接目标节点。
3. 未选定消息仍通过原始传输发送，且不会到达控制器。
4. 控制器已接受但从未转发的消息绝不会被目标处理器观察到。
5. 延迟转发的消息只在回调投递之后才会被观察到。
6. 重排和复制的回调严格按照回调顺序到达原生接收路径，并保留重复消息。
7. 目标处理器收到原始逻辑源身份、路由元数据和已解码的原生消息。
8. 控制器不可用、超时、非 `2xx` 响应、序列化失败以及队列已满绝不会导致回退到直接发送。
9. 格式错误、过大、目标错误、来源未知、路由未知或载荷无效的回调会被拒绝，且不会 panic。
10. 并发提交会为每个有向节点对生成唯一且单调递增的 ID，并且不存在竞态。
11. 启动、部分启动失败、重新连接和重复关闭不会泄漏工作任务、保留过期源映射或 panic。
12. 格式错误的转发消息会被拒绝并记录，而节点保持健康且可以接受之后的合法消息。
13. 由于本地实现缺陷，合法选定消息无法分类、编码、解码或路由时，会触发有序的致命关闭。
14. 控制器注册重试耗尽、就绪后控制器不可用、提交超时、非 `2xx` 提交响应和回调监听器故障，均会产生非零实验失败状态，且不会回退到直接发送。
15. 立即原样转发的控制器能够保持选定消息的原生字节、路由、逻辑来源、重新注入顺序和重复次数与对应普通接收路径一致，预期的介导边界除外。
16. 提交和回调能够观察到同一个 `ID`；重复回调保留该 `ID`、不会被去重，并产生彼此独立的原生重新注入尝试。
17. 测试和诊断信息将回调成功描述为重新注入接受，而不将其视为协议执行或状态改变的证据。
18. 插桩清单可以被机器读取，并且与选定消息清单、生成钩子、已修改符号和记录的验证结果一致。

测试中允许使用仅供测试的轻量级伪控制器。本任务 **MUST NOT** 实现生产控制器、调度策略、消息数据库或用户界面。

## 12. 完成标准

- 目标规范指定的格式化、静态检查和竞态检查均通过；并且

- 最终报告将每个已修改函数映射到规范条款，并列出清单位置或内嵌 JSON、命令、结果、假设、未解决风险以及任何范围扩展请求。
