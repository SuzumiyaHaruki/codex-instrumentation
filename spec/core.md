# Controller-Mediated Message Delivery: Core Specification

## 1. Purpose

This specification defines a reusable instrumentation mode for distributed systems. In this mode, selected node-to-node messages are not delivered through their original direct network path. The sending node submits them to an external central controller, and a selected message may enter the destination's native receive path only if and when the controller sends it to the destination's callback endpoint.

The controller may deliver, delay, reorder, duplicate, or permanently drop a submitted message. Implementing the controller or any controller-side policy is outside the scope of the node changes.

The primary deliverable governed by this specification is node-side instrumentation generated by an LLM-based or equivalent automated implementation agent from requirements and source analysis. The implementation scope includes source discovery, patch generation, controller submission, callback reception, native reinjection, and verification. It excludes controller policy, model-guided scheduling, protocol-state observation, formal-model mapping, and coverage guidance.

This file is system-independent. It defines the stable semantic contract, wire contract, failure model, implementation evidence, and conformance obligations shared by every supported system. A `target-*.md` specification binds that contract to one target system. Both this file and the applicable target specification are normative. The resulting instrumentation SHOULD remain usable by a future external scheduler or model-guided testing system without requiring such a system to be implemented as part of the instrumentation task.

## 2. Normative Language

The terms **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, and **MAY** are to be interpreted as described in RFC 2119 and RFC 8174.

Requirements in this file use stable identifiers of the form `CORE-*`. A target specification may strengthen a core requirement, but it MUST NOT weaken or contradict one.

## 3. Specification Suite Structure

The specification suite has two normative layers:

| Layer | Responsibility | Stability |
| --- | --- | --- |
| **Core specification** | Defines system-independent mediation semantics, controller wire behavior, failure classes, safety constraints, and common conformance properties. | Reused unchanged across target systems. |
| **Target specification** | Defines the target-system vocabulary, selected-message domain, repository-analysis obligations, native binding contract, and target-specific conformance evidence. | Authored for a system family and resolved against the checked-out revision. |

A target specification is a realization profile, not a version-specific patch recipe. It states what the implementer MUST discover, preserve, bind, and prove. Concrete files, symbols, and registered message types are derived from the source revision unless they are stable parts of the target's public architecture.

Every `target-*.md` specification MUST use the following top-level profile. Additional subsections are allowed, but the responsibilities of these sections MUST remain distinct and MUST appear in this order.

| Section | Required content |
| --- | --- |
| **1. Specification Profile** | Target identity, core dependency, requirement namespace, revision policy, and normative status. |
| **2. Semantic Scope** | Target-specific objective, protocol or subsystem vocabulary, protected semantics, and explicit non-goals. |
| **3. Repository Binding Analysis** | Evidence-backed obligations for discovering send, receive, identity, routing, codec, lifecycle, bypass paths, and the machine-readable instrumentation manifest. |
| **4. Controlled Message Domain** | Semantic message families that are selected or excluded, plus the authoritative method for deriving the concrete inventory. |
| **5. Target Binding Contract** | Mapping of core identities, routes, payloads, type labels, and native delivery onto target abstractions. |
| **6. Integration Constraints** | Placement, locality, compatibility, concurrency, and dependency constraints. |
| **7. Configuration and Lifecycle** | Enablement, readiness, registration, startup, shutdown, and resource ownership. |
| **8. Failure Mapping** | Classification of target-specific failures into the core error taxonomy. |
| **9. Conformance Verification** | Clause-linked static, unit, integration, race, and regression evidence. |
| **10. Acceptance Traces** | End-to-end observable scenarios for delivery, omission, reordering, duplication, malformed input, infrastructure failure, and ordinary traffic. |

- **CORE-PROFILE-001:** A target profile MUST identify its target system or system family, applicable core specification, requirement namespace, and source-revision policy before stating implementation requirements.
- **CORE-PROFILE-002:** Target-specific normative requirements MUST have stable identifiers. Explanatory protocol background MAY remain unnumbered but MUST NOT silently introduce an implementation obligation.
- **CORE-PROFILE-003:** The Semantic Scope MUST use the target system's native vocabulary while distinguishing protocol-level concepts from version-specific implementation names.
- **CORE-PROFILE-004:** The Repository Binding Analysis MUST require source-level evidence and MUST produce an integration map, a concrete selected-message inventory, and a bypass-risk inventory before production edits are finalized.
- **CORE-PROFILE-005:** The Controlled Message Domain MUST define selection primarily by semantic ownership or protocol role. Historical numeric identifiers or type lists MAY be documented as orientation but MUST NOT replace authoritative discovery from the current revision.
- **CORE-PROFILE-006:** The Target Binding Contract MUST explain how each core identity, routing, serialization, callback, and native-delivery concept maps to the target system without redefining native message semantics.
- **CORE-PROFILE-007:** Failure Mapping MUST classify target-specific failures under Section 5.4. It MUST NOT invent a recovery behavior that weakens the core no-fallback or fatal-integrity rules.
- **CORE-PROFILE-008:** Conformance Verification MUST link target requirements to observable evidence. A code change without a corresponding clause and verification obligation is incomplete.
- **CORE-PROFILE-009:** Repository Binding Analysis MUST produce a machine-readable instrumentation manifest that records the discovered bindings and generated changes defined by `CORE-PATCH-006` through `CORE-PATCH-009`. The manifest is evidence for automation and downstream adapters; it MUST NOT replace source inspection or conformance testing.

## 4. Terminology

| Term                           | Meaning                                                                                            |
| ------------------------------ | -------------------------------------------------------------------------------------------------- |
| **original transport**   | The system's existing direct node-to-node message path.                                            |
| **controller**           | The external service that receives selected messages and decides whether and when to forward them. |
| **instrumented node**    | A node running with controller-mediated delivery enabled.                                          |
| **selected message**     | A message classified for controller mediation by the target specification.                         |
| **unselected message**   | Any message not classified as selected.                                                            |
| **source identity**      | The stable network identity of the logical sender.                                                 |
| **destination identity** | The stable network identity of the intended receiver.                                              |
| **callback endpoint**    | The HTTP endpoint on an instrumented node that accepts controller-forwarded messages.              |
| **native receive path**  | The existing internal handler invoked after ordinary network decoding.                             |
| **reinjection acceptance** | Successful validation and handoff of a controller-forwarded message to the native receive path or its bounded input queue; it does not imply protocol execution or a state change. |
| **instrumentation manifest** | A machine-readable record of the source bindings, selected-message inventory, generated hooks, configuration, and verification evidence for one instrumentation result. |

## 5. Required Architecture

For every selected message, the observable path SHALL be:

```text
source node
    -> controller submission endpoint
    -> controller policy decision
    -> destination node callback endpoint (only if delivered)
    -> destination native receive path
```

The prohibited path is:

```text
source node -> destination node through the original transport
```

Unselected messages continue to use the original transport.

### 5.1 Operating mode

- **CORE-MODE-001:** Instrumentation MUST be explicitly opt-in and disabled by default.
- **CORE-MODE-002:** With instrumentation disabled, message routing, startup, shutdown, configuration defaults, return values, and error behavior MUST be unchanged.
- **CORE-MODE-003:** Enabling instrumentation MUST NOT change the distributed protocol's validity rules, quorum rules, state transitions, message schemas, persistent formats, or timeout semantics.
- **CORE-MODE-004:** The implementation MUST NOT contain controller policy. A node only submits selected messages and accepts controller-forwarded messages.

### 5.2 Complete interception

- **CORE-INT-001:** The target specification MUST define the selected-message set unambiguously.
- **CORE-INT-002:** Every outbound API capable of sending a selected message, including blocking, non-blocking, broadcast, retry, and reply paths, MUST be covered. Wrapping only one send method is insufficient if another method can bypass it.
- **CORE-INT-003:** Once a message is classified as selected, the node MUST NOT send that message through the original transport under any circumstance.
- **CORE-INT-004:** Encoding failure, a full instrumentation queue, controller unavailability, a timeout, a non-success HTTP response, or shutdown MUST be handled according to Section 5.4 and MUST NOT cause fallback to the original transport.
- **CORE-INT-005:** A selected message MUST NOT enter the destination's native receive path until the destination callback endpoint has accepted that controller-forwarded instance.
- **CORE-INT-006:** While instrumentation is enabled, a selected message that arrives through the original network receive path MUST NOT be delivered to the native receive path. It MUST be rejected or ignored with diagnostics.
- **CORE-INT-007:** Unselected messages MUST continue through the original transport and MUST NOT be submitted to the controller.

### 5.3 Send result semantics

- **CORE-SEND-001:** A blocking send of a selected message MAY wait only within the system's existing send-time bound or a stricter configured bound.
- **CORE-SEND-002:** A non-blocking send of a selected message MUST remain non-blocking. It SHOULD enqueue into a bounded controller-outbound queue and return failure immediately when that queue cannot accept the message.
- **CORE-SEND-003:** A successful send result means that the instrumentation layer accepted responsibility for submission; it does not mean that the controller will deliver the message.
- **CORE-SEND-004:** A selected-message send failure that is known before acceptance MUST be surfaced through the existing return or error mechanism and recorded in diagnostics. It MUST NOT be converted into a direct send.
- **CORE-SEND-005:** If submission fails after the original call returned success, the instrumentation run is no longer trustworthy and the node MUST enter the fatal-failure procedure defined by `CORE-ERR-008`.

### 5.4 Error classification and required action

The implementation MUST distinguish a bad individual message from a broken instrumentation path. Silently continuing after infrastructure or invariant failure would make experiment results unreliable; terminating the process for every malformed input would make the callback endpoint unnecessarily fragile.

- **CORE-ERR-001:** Every instrumentation error MUST be classified as exactly one of: message-scoped rejection, synchronous send rejection, local invariant failure, or controller/control-channel failure.
- **CORE-ERR-002:** A malformed, oversized, wrong-destination, unknown-source, unknown-route, or otherwise invalid controller-forwarded message is a message-scoped rejection. The node MUST reject or drop only that instance, return an appropriate non-`2xx` response, emit a structured error and metric, and continue running.
- **CORE-ERR-003:** Bounded local backpressure or a send attempted after orderly shutdown has begun, when reported to the caller before the message is accepted, is a synchronous send rejection. The send MUST return failure, record the event, leave the node running or continue its existing shutdown, and MUST NOT use the original transport.
- **CORE-ERR-004:** Failure to classify or encode a valid selected outbound message, absence of a required local decoder or native route for a valid selected message, internal queue corruption, or violation of an interception invariant is a local invariant failure. The node MUST treat it as fatal rather than silently dropping the message.
- **CORE-ERR-005:** Failure of the required controller path after startup readiness, including connection refusal, request timeout, malformed HTTP behavior, non-`2xx` submission response, or unexpected callback-listener termination, is a controller/control-channel failure and MUST be treated as fatal.
- **CORE-ERR-006:** Registration MAY use bounded retries before the node declares itself ready. If that retry budget is exhausted, startup MUST fail. A node MUST NOT join the experiment in an unregistered or partially ready state.
- **CORE-ERR-007:** Any asynchronous loss discovered after a selected send has been reported successful MUST be treated as fatal, because the caller can no longer observe or compensate for the loss.
- **CORE-ERR-008:** Fatal handling MUST atomically mark instrumentation unhealthy, stop accepting new selected sends, record the exact cause, initiate orderly node shutdown, and make the owning process return a non-zero failure status. Library code SHOULD return or propagate a typed fatal error; it MUST NOT use an uncontrolled panic or abrupt process exit when orderly shutdown is possible.
- **CORE-ERR-009:** Repeated message-scoped rejections MAY trigger an operator-configured experiment abort threshold, but a single invalid forwarded message MUST NOT terminate the node unless processing it exposes a local invariant failure.

## 6. Controller HTTP Contract

The controller address and the callback listen/advertise address MUST be configuration values. Unless the target specification explicitly states otherwise, addresses are `host:port` values and the node uses plain HTTP in an isolated test network.

All requests in this section MUST use `Content-Type: application/json`.

### 6.1 Node registration

- **CORE-REG-001:** An instrumented node MUST expose its callback endpoint before registering with the controller.
- **CORE-REG-002:** The node MUST register before it can submit its first selected message.
- **CORE-REG-003:** Registration uses `POST http://<controller>/replica` with the following JSON object:

```json
{
  "id": 1,
  "alias": "stable-source-identity",
  "addr": "node-callback-host:port"
}
```

- **CORE-REG-004:** `id` is the experiment-assigned integer node number; `alias` is the stable network identity used in message routing; and `addr` is an address reachable by the controller.
- **CORE-REG-005:** A target adapter MAY add a `metadata` object only when controller routing genuinely requires additional non-secret identity information. The target specification or implementation report MUST document its fields. Private keys and signing secrets MUST NOT be registered.
- **CORE-REG-006:** Registration succeeds only after a `2xx` response. The response body is not semantically significant.
- **CORE-REG-007:** A node MAY retry failed registration only with bounded, shutdown-aware backoff before readiness. Exhausting the retry budget MUST fail instrumentation startup according to `CORE-ERR-006`.
- **CORE-REG-008:** Registration MUST also work for nodes that do not pass through an optional synchronization phase. Readiness MUST be tied to actual callback and send-path readiness, not to one incidental protocol transition.

### 6.2 Submitted and forwarded message

Both directions use the same outer JSON object:

```json
{
  "From": "source-identity",
  "To": "destination-identity",
  "Data": "base64-encoded-target-payload",
  "Type": "target-defined-message-label",
  "ID": "source-identity_destination-identity_0"
}
```

- **CORE-WIRE-001:** The source node submits the object with `POST http://<controller>/message`.
- **CORE-WIRE-002:** The destination node accepts controller-forwarded objects with `POST http://<node-callback>/message`.
- **CORE-WIRE-003:** JSON member names and capitalization MUST be exactly `From`, `To`, `Data`, `Type`, and `ID`.
- **CORE-WIRE-004:** `From` and `To` MUST contain stable network identities, not socket addresses or transient connection indices.
- **CORE-WIRE-005:** `Data` is an opaque byte sequence represented by standard JSON base64 encoding. Its decoded content is defined by the target specification. The controller need not decode or modify it.
- **CORE-WIRE-006:** `Type` is a deterministic, human-readable classifier defined by the target specification. It is controller metadata and MUST NOT replace the target's native routing metadata inside `Data`.
- **CORE-WIRE-007:** `ID` MUST be generated at the source as `<From>_<To>_<counter>`. The counter starts at `0`, increases by one for each submitted message in that directed source/destination pair, and is safe under concurrent sends.
- **CORE-WIRE-008:** The source MUST assign `ID` before submission. The destination MUST preserve it for diagnostics but MUST NOT locally deduplicate it; deliberate controller duplication must remain observable.
- **CORE-WIRE-009:** A submission is accepted only after a `2xx` response; submission failure is handled by `CORE-ERR-005`. A callback returns `2xx` only after the complete outer object and target payload have been validated and safely handed to the native receive path or its bounded local reinjection queue.
- **CORE-WIRE-010:** Malformed requests MUST return `400`. A request that is well-formed but is not valid for the current node or current connection state MUST return a suitable non-`2xx` response, MUST NOT be delivered, and is handled as a message-scoped rejection under `CORE-ERR-002`.
- **CORE-WIRE-011:** `ID` identifies the submitted message instance and MUST be preserved unchanged from source submission through every controller callback. Repeated callbacks for a deliberate controller duplication retain the same `ID`; the node MUST treat each callback as a distinct reinjection attempt and MUST NOT infer a separate delivery identifier by changing `ID`.
- **CORE-WIRE-012:** A callback `2xx` response and reinjection acceptance mean only that instrumentation validation and native handoff succeeded. They MUST NOT be reported or interpreted as proof that a protocol handler completed, accepted the message semantically, changed protocol state, persisted data, or executed a formal-model action.

## 7. Destination-Side Delivery

- **CORE-RECV-001:** The callback handler MUST verify that `To` equals the local stable identity.
- **CORE-RECV-002:** The callback handler MUST validate `From`, `Type`, and all target-defined routing and payload fields before enqueueing the message.
- **CORE-RECV-003:** The implementation MUST resolve `From` to the same logical peer object or source context that the original receive path would have supplied. A message from an unknown or disconnected source MUST NOT be delivered.
- **CORE-RECV-004:** The target payload MUST be decoded with the system's existing codec and wrapper/unwrapper rules. The instrumentation layer MUST not define an alternative semantic message format.
- **CORE-RECV-005:** After successful decoding, the node MUST call the existing native receive path with the original routing metadata, logical source, and decoded message.
- **CORE-RECV-006:** Controller-forwarded delivery MUST pass through the same semantic validation and peer-error behavior as an equivalent message received normally. Malformed controller input MUST reject only that message and MUST NOT panic or stop the process unless it exposes a local invariant failure.
- **CORE-RECV-007:** Messages MUST be delivered in callback-acceptance order. The node MUST NOT restore source-send order, deduplicate, or add an independent delay. Those decisions belong to the controller.
- **CORE-RECV-008:** Queue access, identity lookup, counters, and lifecycle state MUST be concurrency-safe. Queues MUST be bounded or provide equivalent backpressure.
- **CORE-RECV-009:** The instrumentation responsibility ends after invoking the native receive path or safely enqueueing the message for that path. The patch MUST NOT modify protocol handlers to manufacture an execution acknowledgement, infer a state transition, or expose protocol-state mutation solely for future testing integration.
- **CORE-RECV-010:** Structured diagnostics MUST make the same `ID`, `From`, `To`, and `Type` correlatable across submission, callback rejection, reinjection acceptance, and reinjection failure. Diagnostic names MUST distinguish reinjection acceptance from protocol execution and MUST NOT claim an effect that the instrumentation layer cannot observe.

## 8. Lifecycle and Failure Handling

- **CORE-LIFE-001:** Startup order MUST be: validate configuration, bind the callback listener, start delivery processing, register with the controller, then allow selected-message submission.
- **CORE-LIFE-002:** Binding failure, invalid controller configuration, or exhausted registration retries MUST fail instrumentation startup clearly. A required callback or controller path that fails after readiness MUST trigger orderly fatal shutdown.
- **CORE-LIFE-003:** Shutdown MUST stop new submissions, stop accepting callback requests, cancel retry and worker activity, and close the original transport.
- **CORE-LIFE-004:** Shutdown MUST be idempotent and MUST NOT panic when startup was only partially completed.
- **CORE-LIFE-005:** No worker may busy-spin. Delivery SHOULD be event-driven; if polling is unavoidable, it MUST use a reasonable, shutdown-aware interval.
- **CORE-LIFE-006:** HTTP clients MUST have finite dial, response-header, and total request timeouts. Response bodies MUST be closed.
- **CORE-LIFE-007:** Reconnection MUST update the source-identity mapping, and disconnection MUST remove stale mappings so delayed messages cannot be attributed to an obsolete connection.

## 9. Security Boundary

- **CORE-SEC-001:** This mode is for controlled experiments and MUST be clearly documented as unsafe for an untrusted network unless authentication and transport security are added by a target specification.
- **CORE-SEC-002:** The callback listener SHOULD bind to a restricted interface, and network access SHOULD be limited to the controller.
- **CORE-SEC-003:** Registration and message submission MUST NOT expose private keys, signing secrets, authentication tokens, or unrelated node state.
- **CORE-SEC-004:** Logs MUST NOT print secrets or complete sensitive payloads.
- **CORE-SEC-005:** Request body size and decoded payload size MUST be bounded by the target system's existing maximum-message constraints.

## 10. Patch Discipline

- **CORE-PATCH-001:** Changes MUST be minimal and localized around transport, node assembly, configuration, and focused tests.
- **CORE-PATCH-002:** Existing message types, route identifiers, codecs, protocol handlers, and persistent data formats MUST be reused, not replaced.
- **CORE-PATCH-003:** The normal node implementation SHOULD be shared. Large copies of node-construction logic are prohibited when a transport factory, option, or small conditional can provide the instrumentation mode.
- **CORE-PATCH-004:** No production dependency may be added solely to implement the HTTP or JSON portions when the language runtime or an existing dependency already provides them.
- **CORE-PATCH-005:** Existing tests MUST NOT be deleted, skipped, weakened, or rewritten to accommodate the instrumentation.
- **CORE-PATCH-006:** The implementation process MUST produce a machine-readable instrumentation manifest. The current phase prompt determines its writable location; if no manifest path is authorized, an equivalent JSON object MUST be included in the final report rather than written outside scope.
- **CORE-PATCH-007:** The manifest MUST record the target and source revision; selected and explicitly excluded message families; discovered outbound hooks and bypass risks; callback and native-reinjection bindings; identity, route, codec, configuration, and lifecycle bindings; changed files and symbols with specification clauses; verification commands and results; assumptions; and unresolved gaps. Entries that cannot be established MUST be marked unresolved rather than guessed.
- **CORE-PATCH-008:** The manifest MUST contain no secret material, MUST be checked against the generated patch and source evidence, and MUST NOT contain or require controller policy, protocol-state observation, formal-model actions, model states, or fuzzing strategy.

The manifest has the following minimum JSON shape:

```json
{
  "schema_version": 1,
  "target": {
    "name": "target-system",
    "revision": "source-revision"
  },
  "selected_messages": [],
  "excluded_messages": [],
  "bindings": {
    "outbound_hooks": [],
    "bypass_risks": [],
    "callback": {},
    "native_reinjection": {},
    "identity": {},
    "routing": {},
    "codec": {},
    "configuration": {},
    "lifecycle": {}
  },
  "changes": [],
  "verification": [],
  "assumptions": [],
  "unresolved_gaps": []
}
```

- **CORE-PATCH-009:** Manifest producers MAY add target-specific members, but MUST NOT rename, remove, or change the meaning of the minimum members. Every binding and change entry MUST include source evidence or an explicit unresolved marker, and every value MUST be valid JSON data.

## 11. Required Verification

The implementation MUST add focused tests proving all of the following:

1. Instrumentation is off by default and the ordinary path is unchanged.
2. Every selected blocking, non-blocking, broadcast, reply, and retry send path reaches the controller and never the direct destination.
3. Unselected messages still use the original transport and never reach the controller.
4. A controller-accepted but never-forwarded message is never observed by the destination handler.
5. A delayed forwarded message is observed only after callback delivery.
6. Reordered and duplicated callbacks reach the native receive path in exactly callback order, including duplicates.
7. The destination handler receives the original logical source identity, routing metadata, and decoded native message.
8. Controller outage, timeout, non-`2xx` response, serialization failure, and a full queue never cause direct-send fallback.
9. Malformed, oversized, wrong-destination, unknown-source, unknown-route, and invalid-payload callbacks are rejected without panic.
10. Concurrent submission produces unique monotonic IDs per directed pair and is race-free.
11. Startup, partial-startup failure, reconnection, and repeated shutdown do not leak workers, retain stale source mappings, or panic.
12. A malformed forwarded message is rejected and logged while the node remains healthy and can accept a later valid message.
13. A valid selected message that cannot be classified, encoded, decoded, or routed because of a local implementation defect causes orderly fatal shutdown.
14. Controller registration exhaustion, post-readiness controller outage, submission timeout, non-`2xx` submission response, and callback-listener failure all produce a non-zero experiment failure without direct-send fallback.
15. An immediate pass-through controller preserves the selected message's native bytes, route, logical source, reinjection order, and multiplicity relative to the corresponding ordinary receive path, apart from the intended mediation boundary.
16. The same `ID` is observable at submission and callback; repeated callbacks retain that `ID`, are not deduplicated, and produce distinct native reinjection attempts.
17. Tests and diagnostics describe callback success as reinjection acceptance and do not treat it as evidence of protocol execution or state change.
18. The instrumentation manifest is machine-readable and agrees with the selected-message inventory, generated hooks, changed symbols, and recorded verification results.

A lightweight fake controller used only by tests is permitted. A production controller, scheduling policy, message database, or user interface MUST NOT be implemented as part of this task.

## 12. Completion Criteria

The work is complete only when:

- every applicable `CORE-*` clause and every target-specific clause is met;
- all original tests and all new focused tests pass;
- formatting, static checks, and race checks identified by the target specification pass; and
- the instrumentation manifest has been produced and checked against the implementation; and
- the final report maps changed functions to specification clauses and lists the manifest location or embedded JSON, commands, results, assumptions, unresolved risks, and any scope requests.
