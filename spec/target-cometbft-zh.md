# CometBFT 的中央控制器介导消息投递规范

## 1. 规范剖面

本目标实现剖面将[中央控制器介导的消息投递：核心规范](./core-zh.md)应用于 CometBFT，包括源自 Tendermint 协议谱系的版本。本文件与 `core-zh.md` 共同构成规范性要求，目标专属条款使用 `CMT-*` 命名空间。由于不同 CometBFT 版本的包布局、P2P 抽象、消息注册表、Reactor 接口、启动流程和命令结构可能不同，本剖面固定语义义务和证据要求，但不预先规定包布局、符号列表、数值 Channel ID 或补丁形态；所有版本局部的绑定关系都必须从当前检出的源码树中推导。

- **CMT-PROFILE-001:** 实现 **MUST** 从当前检出的源码树中推导具体文件、符号、已注册消息、Channel 描述符和生命周期钩子，并且 **MUST** 保留该版本已经建立的架构和行为。
- **CMT-PROFILE-002:** 如果本规范提到的协议概念在当前版本中已经改名、拆分、合并或移除，实现 **MUST** 记录该版本中的等价对象，或给出有证据支持的“不适用”结论。实现 **MUST NOT** 将历史抽象强行加入代码库。

## 2. 语义范围

- **CMT-SCOPE-001:** 选定的 CometBFT 消息 **MUST** 只能通过控制器提交路径离开发送方，并且 **MUST** 只有在控制器回调投递后，才能到达目标节点的普通 Reactor 或子系统处理器。
- **CMT-SCOPE-002:** 实现范围包括配置、控制器注册、出站拦截、回调接收、原生解码、原生分派、生命周期集成、诊断信息和聚焦一致性测试。
- **CMT-SCOPE-003:** 实现 **MUST NOT** 实现控制器策略、添加改变共识状态的远程命令、修改现有 Protobuf 定义或持久化格式，也不得在插桩禁用时改变普通行为。
- **CMT-SCOPE-004:** 插桩属于通信层干预。介导 `Proposal`、`ProposalPOL` 或 `BlockPart` 流量 **MUST NOT** 改变提案构造、Proposer 选择、提案有效性检查、Locked Value 或 Valid Value 处理，也不得改变从 Propose 到 Prevote 的转换。介导 Prevote 或 Precommit `Vote` 流量 **MUST NOT** 改变投票构造或签名、`nil` 投票语义、Voting Power 计算、Quorum 阈值、决定规则、Height/Round/Step 转换或共识超时。
- **CMT-SCOPE-005:** `NewRoundStep`、`NewValidBlock`、`HasVote`、`VoteSetMaj23` 和 `VoteSetBits` 等 Round-state 与 Vote-set 消息在控制器投递后 **MUST** 仍然作为普通 Consensus Reactor 输入处理。适配器 **MUST NOT** 将它们解释为命令、合成缺失的提案或投票状态，也不得绕过现有 Reactor 处理器修改 Peer 共识状态。
- **CMT-SCOPE-006:** 本剖面约束自动生成的节点侧消息插桩。它不要求生成补丁观测 Proposal、BlockPart、Prevote、Precommit 或其他重新注入消息之后是否改变 Consensus State，不要求发出协议执行确认或将事件/状态映射到 TLA+，也不要求实现模型引导调度、覆盖率、变异或故障策略。

## 3. 代码库绑定分析

实现由分析驱动。以下义务用于建立有证据支持的集成映射，并以该映射约束后续生产代码修改。

- **CMT-ANALYSIS-001:** 识别普通节点使用的配置加载与验证路径、节点构造路径、启动命令路径以及关闭/错误传播路径。
- **CMT-ANALYSIS-002:** 识别所有能够承载选定流量的出站抽象，以及 Consensus Reactor 或 State Sync Reactor 的 Gossip 例程，包括阻塞发送、非阻塞发送、Peer 发送、广播、直接回复、重试和辅助路径。沿实际 P2P 边界追踪原生 `Envelope` 或等价对象；没有证据时，不得假设这些路径最终汇合。
- **CMT-ANALYSIS-003:** 识别从 Channel 和网络字节开始，依次经过路由选择、原生解码、移除包装、Peer 归属、验证，直至最终 Reactor 或子系统分派的普通入站路径。
- **CMT-ANALYSIS-004:** 识别当前版本注册 Channel 描述符、消息原型、编解码器和所属 Reactor 的位置，并确定适配器可以复用哪些注册表。
- **CMT-ANALYSIS-005:** 识别稳定且经过认证的 Node ID、提供给处理器的内存 Peer 对象，以及创建、替换或删除两者映射的所有连接生命周期事件。必须区分 P2P 身份、Validator 地址和投票身份。
- **CMT-ANALYSIS-006:** 在出站和入站两个方向，分别追踪当前版本中代表 Round-state Gossip、Proposal 与 BlockPart 传输、Prevote、Precommit、Vote-set 协调以及每个选定 State Sync 消息族的具体消息。
- **CMT-ANALYSIS-007:** 枚举所有已发现的绕行风险。至少比较 Peer 的阻塞和非阻塞发送、广播展开、接收处理器内部创建的回复以及对更低层 Transport 的直接调用。
- **CMT-ANALYSIS-008:** 在最终报告以及 `CORE-PROFILE-009` 要求的插桩清单中记录所得集成映射、选定消息清单、绕行风险清单以及已修改符号到条款的映射。如果源代码树存在歧义或与本规范矛盾，应停止并报告歧义，而不是猜测。
- **CMT-ANALYSIS-009:** 对每个候选选定原生类型，记录其协议角色、Channel 描述符、所属 Reactor、包装器与编解码器、出站入口、入站处理器、选择状态和源码证据。该绑定表是必需的中间产物，而不是可选的实现说明。
- **CMT-ANALYSIS-010:** 插桩清单还 **MUST** 标明当前版本中的具体回调路由、原生 Reactor 或子系统重新注入入口、稳定 P2P 身份来源、Peer 对象查找生命周期、适配器载荷版本与字段、插桩配置键以及准确验证命令。它 **MUST** 明确标记版本局部的不确定性，并且 **MUST NOT** 包含共识状态提取或形式化模型映射。
- **CMT-ANALYSIS-011:** 识别所有可能在普通注册完成后替换或改变所属 Reactor、Channel 描述符、消息原型、包装器或编解码器的构造期 Option、自定义 Reactor 机制和注册挂钩。绑定结果 **MUST** 区分无关扩展与对选定 Consensus 或 State Sync 路由的覆盖，并且 **MUST** 给出能够在插桩就绪前发现不兼容覆盖的源码证据。

## 4. 受控消息域

- **CMT-SELECT-001:** 选择由 CometBFT Consensus Reactor 负责、用于实现或 Gossip 共识进展的每一种 P2P 消息。具体清单 **MUST** 覆盖当前版本中以下概念的等价类型：`NewRoundStep` 和 `NewValidBlock` 等 Round-state 通知；`Proposal`、`ProposalPOL` 和 `BlockPart` 等提案传播；Prevote 和 Precommit 两类签名 `Vote`，包括 `nil` 投票；以及 `HasVote`、`VoteSetMaj23` 和 `VoteSetBits` 等投票知识或协调消息。
- **CMT-SELECT-002:** 选择由 State Sync Reactor 负责的每一种 P2P 消息，使同步流量无法绕过控制器调度。清单 **MUST** 覆盖当前版本已注册的快照发现、快照通告、Chunk 请求/响应、Light Block 请求/响应和 Consensus Parameters 请求/响应消息族。
- **CMT-SELECT-003:** 从当前版本的 Consensus 和 State Sync Channel 描述符及消息注册表中推导具体的选定路由和原生类型。Consensus State、Data、Vote 和 Vote Set Bits Channel，或其当前版本等价物，均 **MUST** 被纳入核对。历史数值 Channel ID 和上述常见类型名称仅用于定位，不能取代权威注册数据。
- **CMT-SELECT-004:** 仅由交易传播、Evidence 传播、Block Sync、Peer Exchange、RPC 或应用通信负责的流量保持未选定，除非源码分析证明它与选定流量共享路由且无法安全分离。
- **CMT-SELECT-005:** 选定流量的每一种已发现发送模式都 **MUST** 被拦截。更低层路径 **MUST NOT** 继续作为选定消息调用方的意外绕行通道。
- **CMT-SELECT-006:** 在插桩模式下，通过普通直接 P2P 接收路径到达的选定流量 **MUST** 在原生分派前被拒绝。插桩禁用时，同一路径 **MUST** 保持不变。

## 5. 目标绑定契约

### 5.1 外层控制器契约

- **CMT-ADAPTER-001:** 使用 `core-zh.md` 中的注册和外层消息 HTTP 契约，不得改变成员名称或含义。
- **CMT-ADAPTER-002:** 使用节点稳定且经过认证的 P2P 身份作为 `alias` 和 `From`；使用经过认证的目标身份作为 `To`。不得使用套接字地址、Moniker、Validator 地址或临时 Peer 索引作为路由身份。
- **CMT-ADAPTER-003:** 除非源码分析证明需要额外的非秘密元数据，否则注册 **MUST** 只包含核心规范中的 `id`、`alias` 和回调 `addr` 字段。私钥和签名材料 **MUST NOT** 被发送。

### 5.2 不透明目标载荷

- **CMT-ADAPTER-004:** `Data` **MUST** 包含足以在控制器无需解释的情况下重建原始 CometBFT 路由和原生序列化消息的适配器载荷。
- **CMT-ADAPTER-005:** 适配器载荷模式 **MAY** 选择适合当前版本的形式，但它 **MUST** 是确定的、版本局部的、有界的，并在聚焦测试和最终报告中说明。它 **MUST** 包含原生路由标识符和准确的原生消息字节。
- **CMT-ADAPTER-006:** 使用普通出站 P2P 路径所用的相同 Protobuf 实现和包装行为进行序列化。不得发明第二种具有语义的消息表示。
- **CMT-ADAPTER-007:** 回调投递时，通过现有 Channel/消息注册表解析路由，实例化已注册的原生消息类型，使用普通编解码器解码，并在分派前执行普通解包装操作。
- **CMT-ADAPTER-008:** `Type` **MUST** 是从具体原生消息派生的确定性、人类可读标签。源码分析 **MUST** 为选定清单建立完整映射。遇到没有标签的合法选定出站类型属于 `CORE-ERR-004` 下的本地不变量故障。
- **CMT-ADAPTER-009:** 回调解码后，根据具体消息重新计算标签，并要求其与外层 `Type` 匹配。不匹配属于消息级拒绝。

### 5.3 原生投递语义

- **CMT-ADAPTER-010:** 接受回调投递前，将外层 `From` 解析为当前已认证 Peer 对象。未知、已断开或过期的来源属于消息级拒绝。
- **CMT-ADAPTER-011:** 通过普通 P2P 解码后使用的同一个所属 Reactor 或子系统接收入口投递，并提供相同的路由标识符、具体消息和逻辑源 Peer。
- **CMT-ADAPTER-012:** 复用普通语义验证和 Peer 错误报告。无效控制器输入 **MUST NOT** 导致进程 panic。
- **CMT-ADAPTER-013:** 不得在本地恢复发送顺序、去重、延迟或推断控制器已经投递。回调接受顺序即投递顺序。
- **CMT-ADAPTER-014:** 在从提交到回调处理的结构化诊断信息中原样保留外层 `ID`。控制器针对同一 Proposal、BlockPart、Vote 或其他选定实例进行重复回调时保留相同 `ID`；除非某一次回调按照第 8 节被单独拒绝，否则每次回调都 **MUST** 到达原生重新注入边界。
- **CMT-ADAPTER-015:** 选定消息完成验证和解码、关联到已认证源 Peer，并安全交给普通 Reactor 或子系统接收入口或其有界输入队列后，回调即成功。该成功 **MUST** 被描述为重新注入接受，并且 **MUST NOT** 声称 Consensus Reactor 已经处理该消息、推进 Height/Round/Step、改变 VoteSet、接受 Proposal 或持久化 State Sync 数据。
- **CMT-ADAPTER-016:** 即使多个协议角色共享同一个具体原生消息类型，`Type` 也 **MUST** 保留控制器调度所需的每一项协议角色区别。尤其是，由原生 `Vote` 消息承载的 Prevote 与 Precommit **MUST** 根据签名投票类型生成不同标签；`nil` 投票也 **MUST** 保留这一差别。适配器 **MUST NOT** 将两者折叠为通用 `Vote` 标签，也不得要求控制器解析 `Data` 才能区分它们。

## 6. 集成约束

- **CMT-INTEGRATION-001:** 将出站拦截放在源码分析发现的最窄共享边界上；在该边界处，目标身份和原生消息均已可用，并且能够覆盖选定消息的每种发送模式。
- **CMT-INTEGRATION-002:** 保留普通 P2P Transport，用于连接认证、Peer 生命周期和未选定流量。插桩 **MUST** 与现有 Transport 组合，而不能替换无关的网络行为。
- **CMT-INTEGRATION-003:** 每个连接方向创建的 Peer 都 **MUST** 参与拦截和来源查找。连接清理和替换 **MUST** 删除过期映射。
- **CMT-INTEGRATION-004:** 复用普通节点构造路径。应优先使用 Transport 工厂、依赖选项或小型条件分支，而不是复制节点构造器或 Reactor 设置。
- **CMT-INTEGRATION-005:** 除非某条款明确要求局部改动，否则保留现有连接过滤器、Peer 限制、地址簿行为、Persistent Peer 行为、日志设置、指标设置和服务顺序。
- **CMT-INTEGRATION-006:** 回调处理器 **MUST** 验证消息并将不可变消息入队；在持有 HTTP、队列或 Peer 映射锁时，不得调用 Reactor。
- **CMT-INTEGRATION-007:** 使用标准库或现有 HTTP 工具。不得仅为控制器通信新增生产 Web 框架依赖。
- **CMT-INTEGRATION-008:** 不得添加改变协议的回调端点，也不得修改共识转换来发出就绪信号。就绪状态属于插桩 Transport 或节点生命周期。
- **CMT-INTEGRATION-009:** 结构化插桩诊断信息 **MUST** 使用核心消息字段暴露提交、回调拒绝、重新注入接受和重新注入失败。诊断信息 **MUST** 位于 Consensus State 之外，并且 **MUST NOT** 要求修改 Reactor 状态转换。
- **CMT-INTEGRATION-010:** 在所有构造期 Option 和自定义 Reactor 替换都已应用之后、插桩就绪之前，验证每个选定 Channel 仍由预期的原生 Consensus 或 State Sync Reactor 所有，并使用兼容的已注册原型、包装器和编解码器。仅仅为某个数值 Channel 找到非空 Reactor 和原型并不充分。无法证明兼容的选定路由覆盖 **MUST** 作为 `CORE-LIFE-002` 下的绑定故障使启动失败；不拥有也不改变选定路由的无关自定义 Reactor **MUST** 保持可用。

## 7. 配置与生命周期

- **CMT-LIFE-001:** 使用当前版本已有的命名和验证惯例，添加显式且默认关闭的插桩配置。
- **CMT-LIFE-002:** 配置 **MUST** 提供回调监听/通告地址、控制器地址以及实验分配的整数节点 ID。启用后的配置无效或不完整时，节点 **MUST** 在加入实验前失败。
- **CMT-LIFE-003:** 将启用逻辑集成到普通启动路径，或集成到委托给普通启动路径的轻量命令包装器中。不得创建第二份独立维护的节点实现。
- **CMT-LIFE-004:** 如果仓库包含测试网生成工具，应对其进行最小扩展，使生成节点获得唯一回调地址和 ID 以及公共控制器地址。不得复制生成器。
- **CMT-LIFE-005:** 在允许选定消息发送前启动回调监听器和投递工作任务，并成功完成注册。对于立即开始运行的节点和先执行同步的节点，该行为都 **MUST** 正常工作。
- **CMT-LIFE-006:** 关闭过程 **MUST** 取消控制器请求和工作任务、停止接受回调、关闭现有 Transport，并且在部分启动后仍保持安全。
- **CMT-LIFE-007:** 队列为空时 **MUST** 阻塞等待事件或使用合理定时器；任何插桩工作任务都不得忙等。

## 8. 故障映射

- **CMT-ERROR-001:** JSON 无效、适配器载荷无效、回调提供的路由未知、目标错误、来源过期、载荷过大、所提供字节导致原生解码失败或 `Type` 不匹配，均属于 `CORE-ERR-002` 下的消息级拒绝。拒绝并记录该实例，然后继续运行。
- **CMT-ERROR-002:** 已注册为选定的路由缺少本地解码器或所属处理器、合法选定出站消息无法编码或标记，或者内部适配器状态损坏，均属于 `CORE-ERR-004` 下的本地不变量故障，并且 **MUST** 通过有序的致命关闭停止实验。
- **CMT-ERROR-003:** 控制器注册重试耗尽、就绪后控制器连接失败、提交超时、非 `2xx` 提交响应、提交响应行为异常或回调监听器意外故障，均按照 `CORE-ERR-005` 视为致命故障。
- **CMT-ERROR-004:** 只有当非阻塞本地提交队列已满这一失败在接受消息前已被报告时，才 **MAY** 返回普通发送失败并让节点继续运行。成功返回后的任何消息丢失都是致命故障。
- **CMT-ERROR-005:** 致命错误 **MUST** 到达节点所有者或命令运行器，使进程在有序清理后以非零状态退出。深层 Transport 或 HTTP 代码 **MUST NOT** 隐藏错误、不可控地 panic 或突然退出进程。

## 9. 一致性验证

一致性测试计划从集成映射推导，而不是从假定的符号名称推导。

- **CMT-TEST-001:** 从集成映射推导测试计划，并添加选定路由清单测试，将适配器的选择结果与当前 Consensus 和 State Sync 注册进行比较。
- **CMT-TEST-002:** 对每种已发现的出站发送模式，证明选定消息到达伪控制器且绝不调用直接 Transport，同时代表性的未选定消息保留普通路径。
- **CMT-TEST-003:** 证明适配器载荷中的每个选定具体消息族都能够通过当前原生包装器、编解码器、注册表和解包装行为完成往返。
- **CMT-TEST-004:** 证明回调投递以原始逻辑 Peer 和路由到达正确的现有处理器，包括有意进行的重排和复制。
- **CMT-TEST-005:** 证明只有在插桩模式下，选定的直接 P2P 入站流量才会被拒绝。
- **CMT-TEST-006:** 证明错误分类：单条格式错误的回调会被丢弃且节点保持可用；本地不变量故障和控制器/控制通道故障会导致有序的非零退出，且不会回退到直接发送。
- **CMT-TEST-007:** 在支持的情况下使用竞态检测器测试接受和拨出的 Peer、重新连接、过期消息拒绝、并发 ID、有界队列、部分启动和重复关闭。
- **CMT-TEST-008:** 对每个已修改包运行仓库现有的聚焦格式化、单元测试、静态分析和竞态检查命令。从当前仓库中发现命令，而不是假定历史包路径，并记录准确命令和结果。
- **CMT-TEST-009:** 一致性测试集 **MUST** 至少覆盖一条 Round-state 消息、一组 Proposal 与 BlockPart 流程或其当前版本等价物、一条 Prevote、一条 Precommit、一条 Vote-set 协调消息以及每个选定 State Sync 消息族。每个用例 **MUST** 标明 `CMT-ANALYSIS-009` 发现的原生类型和 Channel。
- **CMT-TEST-010:** 轻量级伪控制器 **MAY** 仅供测试使用。生产补丁 **MUST NOT** 添加控制器可执行文件、调度器、持久化层或策略引擎。
- **CMT-TEST-011:** 使用立即原样返回每条已提交消息的伪控制器，在插桩边界证明 pass-through 等价性：原生路由、消息字节、解码后的具体类型、逻辑源 Peer、重新注入顺序和重复次数 **MUST** 与对应普通 P2P 接收行为一致。
- **CMT-TEST-012:** 证明相同的外层 `ID` 能够关联提交和回调诊断信息，重复回调保留该 `ID` 且不会在本地去重，并且成功回调响应只被报告为重新注入接受，而不是 Consensus 执行或状态改变。
- **CMT-TEST-013:** 根据选定 Channel 和消息清单、已发现发送与接收钩子、已修改文件和符号、配置绑定以及记录的命令结果验证插桩清单。清单过期、不完整或相互矛盾时，一致性验证失败。
- **CMT-TEST-014:** 证明控制器可见标签能够区分 Prevote 与 Precommit，并同时覆盖非 `nil` 和 `nil` 投票。如果重新计算得到的投票角色标签与外层 `Type` 不匹配，回调验证 **MUST** 拒绝该消息，且不得要求控制器解码 `Data`。
- **CMT-TEST-015:** 覆盖构造期扩展兼容性。将拥有选定路由的 Consensus Reactor 或 State Sync Reactor 替换为未经证明兼容的自定义 Reactor 时，节点 **MUST** 在就绪前启动失败；添加不拥有也不改变选定路由的自定义 Reactor 时，启用模式启动和普通行为 **MUST** 保持正常。

## 10. 验收轨迹

### 场景 A：受控丢弃

发送方报告已将一条选定 Prevote 成功提交给控制器，控制器不调用目标节点，并且目标节点绝不会为该投票调用原生 Consensus Reactor 接收入口。

### 场景 B：受控重排与复制

控制器按照选定顺序向目标节点转发 Proposal、BlockPart 和 Precommit 实例，并重复一条 Precommit。目标节点严格按照回调顺序和数量调用现有 Consensus Reactor 接收入口或为其入队，并将经过认证的原始 Peer 作为来源。本场景不声称每个实例都会改变 Consensus State。

### 场景 C：格式错误的转发消息

目标节点拒绝并记录一条格式错误的回调，保持运行，并随后接受一条合法回调。

### 场景 D：控制器故障

就绪后，控制器提交超时或返回非成功状态。不会发生直接发送；插桩被标记为不健康，节点干净关闭，并且实验进程以非零状态退出。

### 场景 E：普通流量

未选定的交易、Evidence、Block Sync 或 Peer Exchange 消息使用现有 P2P 路径，并保留原始返回、错误和接收行为。

### 场景 F：立即原样转发

控制器立即转发每一条选定的 NewRoundStep、Proposal、BlockPart、Prevote、Precommit、Vote-set 或 State Sync 消息，并且不改变外层对象或适配器载荷。目标节点以普通 P2P 路径本应提供的相同路由、解码消息、逻辑 Peer、顺序和重复次数到达同一个原生接收边界。回调成功只表示重新注入接受。
